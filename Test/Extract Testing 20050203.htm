Extract Utility Testing Enhancements

This document supports the testing of the following enhancements/corrects to the Data Extract Utility.

Extract Journaling

Enhancement that collects the changes made to Extract Cursors and Views and stores them in a journaling table.  Collects and stores a required comment from the user related to the change.

 

Summary of changes

1.      The changes to both cursor and view screens

a.       Added new buttons Journal and Comment.

b.      Added new Journal screens to application.

c.       Added application menu to the cursor form (miscellaneous correction).

2.      Changes to the existing process

a.       Updates to data on either the cursor or the view screens, require the user to add a comment.  Comment is saved with Journal.

b.      Deleting a record from either the cursor or the view screens requires the user to add a comment.  Comment is saved with Journal.

3.      Journaling Screen

a.       Allows for viewing the journal information in separate window.

b.      Allows viewing of previous journal entries.

4.      Database changes

a.       Added new column to cursor table NOTE    VARCHAR2(2000)

b.    Added new column to view table NOTE    VARCHAR2(2000)

c.  Created journaling table CT_EXT_CRS_CTL$JN

d.      Created journaling table CT_EXT_VW_CTL$JN

e.       Added synonyms for the new journal tables

f.        Granted the INSERT privilege to the following roles:

EXTRACT_ADMIN

EXTRACT_EXECUTE

EXT_ROLE

Ignored Updates

This enhancement contains corrections that prevent the exclusion of partial records from an update transaction record.

 

Summary of changes

For all non-delete records, the key for each record used to check the data of the last set of extract records.  If there is at least one record of the key where the data does not match the last extracted data, then all records of the key are included into the current extract.  If the records match exactly, then all records of the key are not included in the current extract (deleted).  Records of the extract having the same key are given the same change date using the maximum change date, with in the current set of extract records, for that key.

Delete Dates

This enhancement contains corrections that assign date information related to record creation, as well as extraction date and user to delete records.

 

Summary of changes

For all delete records within the current extract, the maximum create date and change date/time is collected from the archive data using key identifiers and applied to the delete record of the current extract.  Delete records are removed from the current extract where they already exist in the archive identified by key fields and having the same change date.

Data Extract Form

An enhancement has added that allows for the reporting of the current status of an extraction when using the Refresh button from the Main Extraction screen.

 

Summary of changes

The Refresh button was enhanced to display a more accurate summary status of the extraction.  While any step within the extraction process is in Progress, that status will be reflected in the Summary Status field.  When the process has completed, a final status of Success or Failure will  appear within the summary field.

Refresh Extract View Form

This enhancement allows the Current study to follow the user from the Main Extraction screen to the Refresh View screen.

Ignored Updates

Patrick Conrad performed testing on February 8, 2005.  Testing was performed on the OCDEV database and middle tier.  Below are the testing results for Journaling Testing.

 

Scripts and Useful SQL

Identify Script:            Used to identify target study where Ignored Updates exist. 

SELECT protocol, file_id, count(*) Count

  FROM IGNORED_UPDATES_VW

  group by protocol, file_id;

 

Combine Script:         Combines data from all source points, include Archive, Current Extract, Last Extract and Ignored Updates tables/Views:

SELECT protocol, rectype Rtype, file_id, Extracted, changed, pt, del_flag,

       key1, key2, key3, key4, key5, data, data all_data

  FROM IGNORED_UPDATES_VW

 where protocol = '&study'

  and rectype ='ignore_update'

  and file_id = '&file'

Union All 

SELECT protocol, 'prot_arch' Rtype, file_id, extracted, substr(line_text, 32,14) changed, pt, del_flag,

       key1, key2, key3, key4, key5, substr(line_text,95) data, line_text all_data

  FROM CT_DATA a

 where submission_flag = 1

   and a.protocol = '&study '

   and a.file_id = '&file ' 

union all

SELECT protocol, 'prot_crrnt' Rtype, file_id, extracted, substr(line_text, 32,14) changed, pt, del_flag,

       key1, key2, key3, key4, key5, substr(line_text,95) data, line_text all_data

  FROM CT_EXT_DATA a

 where submission_flag = 1

   and a.protocol = '&study '

   and a.file_id = '&file ' 

union all

SELECT protocol, 'prot_last' Rtype, file_id, extracted, substr(line_text, 32,14) changed,

       pt, del_flag, key1, key2, key3, key4, key5, substr(line_text,95) data, line_text all_data

  FROM CT_DATA_LAST_EXT a

 where submission_flag = 1

   and a.protocol = '&study '

   and a.file_id = '&file ' 

order by pt, file_id, key1, key2, key3, rtype, extracted;

 

 

Item

Steps

Status

Identify Ignored Updates

Using the Identify script from above, identify target study, one where there are several Ignored Update Records.

Success

Verify Ignored Updates

Using the Combine script from above, compare the ignored_update records against the prot_arch, prot_crrnt and prot_last records.  In most cases, and ignore update record is one section of a multiple key record where only one record of the multiple record key update is reported, and the other part is reported at a later time.  These records then alternate between each other up to the current reporting period.

Success

Execute Extract

1.      Perform an Extract for the Study Identified above, using the Data Extract Utility.  As part of the Extract process, the Last Extract may have to be removed.

2.      Use the Combine script again, and ensure that a complete set of multiple key records are included in the current extract, identified by the prot_crrnt RTYPE.

NOTE: While performing this test, the Refreshing the Data Extract Form testing can be performed.

Success

Data Extract Form

Patrick Conrad performed testing on February 8, 2005.  Testing was performed on the OCDEV database and middle tier.  Below are the testing results for Journaling Testing.

 

Item

Steps

Status

Perform Extract

The Refresh enhancement allows to the current process status to be displayed in the Extract Status section of the Extract Form.  Every time the refresh button is selected, the Status area displays the most recent status of each step executed.  The summary field displays the current summary status.  If any step is in Progress this value will be displayed.  Until the process is finished, this value will remain.  Once the process copletes, the field will display either a Success message or a Failure message.  A success message signifies that every step of the process was successful, while a failure message signifies that at least one step of the entire process did not complete as expected.  Ensure that repeatedly pressing the Refresh button during an extract will progress from a Progress summary status to either a Success or Failure summary status.  The actual status can be check be scrolling through the individual step statuses.

Success
